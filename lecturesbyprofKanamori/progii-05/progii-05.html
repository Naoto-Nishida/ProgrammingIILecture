分割コンパイル

これまではプログラムのソースコードをひとつのファイルに書いてきたが、今回は、処理ごとに別のファイルに分けて書き、コンパイルするようにしよう。 なぜわざわざファイルを分けて書くかというと、それは次のような理由による。
  • コード量が多くなると、ひとつのファイルではメンテナンスしづらい。 規模が大きくなると、プログラムが数千行、数万行となることはよくある。少し書き直すたびにすべてコンパイルしなおすのは効率が悪い。
  • ソースコードを再利用しやすくなる。 変更されにくい処理を別のファイルに分けておけば、自分用のライブラリとして再利用しやすい。
  • 複数の人で開発するとき、ひとつのファイルだとやりにくい。 手分けして開発するなら別々のファイルにしておいた方がやりやすい。
ソースコードを別々のファイルに分けておくと、ファイルごとにコンパイルすることができる。最終的にプログラムを作るときには、別々にコンパイルしたものを組み合わせる (リンクする)。このように、複数のファイルにソースコードを分割してプログラムをコンパイルすることを、分割コンパイルという。

複数ファイルのコンパイル

例えば、ソースコードとして prog1.c, prog2.c があるとする。これらをまとめてひとつのプログラム prog を作りたい。コンパイルの仕方は 2 通りある。ひとつは、一度にすべてコンパイルする方法である。 
$ cc -o prog  prog1.c  prog2.c
ただしこの方法だと、どれかひとつのファイルに変更を加えただけでも、すべてのファイルを入力としてコンパイルし直さなければいけない。 もうひとつの方法として、ファイルをひとつずつ、別々にコンパイルすることができる。コンパイラに -c というオプションをつけると、コンパイルの中間生成物が作られる。UNIX 環境では拡張子が .o というバイナリファイルである。これをオブジェクトファイルという。ファイルごとにコンパイルが済んだら、オブジェクトファイルや、必要なライブラリ (数学ライブラリやグラフィクスライブラリなど) を結合する。この結合処理をリンクという。例えば数学ライブラリを使うとしたら、次のようなコマンドを実行する。 
$ cc -c prog1.c
$ cc -c prog2.c
$ cc -o prog  prog1.o  prog2.o  -lm
しかし、ファイルを修正するたびに、毎回コマンドを入力するのは面倒である。それを自動化してくれるものとして、UNIX 環境には make コマンドというものが用意されている。

make コマンド

make コマンドを使うためには、Makefile という名前のファイルを、ソースコードと同じディレクトリに置いておく。Makefile の中身は、例えば次のように書く。※インデント (字下げ) しているところはスペースではなく必ずタブ文字にすること。 
# この行はコメント。# で始まる行はコメントになる

prog: prog1.o prog2.o
	cc -o prog prog1.o prog2.o -lm

prog1.o: prog1.c
	cc -c prog1.c

prog2.o: prog2.c
	cc -c prog2.c
このような Makefile を置いておき、make コマンドを実行すると、make コマンドが更新されたファイルの依存関係を調べ、必要なファイルだけを再コンパイルしてくれる。この Makefile の書き方は、基本的には次のようになっている。 
ターゲット:  ターゲットの依存するもの
	ターゲットを作るためのコマンド
これら、「ターゲット」「ターゲットの依存するもの」「ターゲットを作るためのコマンド」がひとつのルールになる。ただし、
  • 「ターゲット」や「ターゲットの依存するもの」には複数のファイル名を書ける。
  • 「ターゲット」が存在しなかったり「ターゲットの依存するもの」よりも古いときは、「ターゲットを作るためのコマンド」を実行して「ターゲット」を作り直す。
  • 「ターゲットを作るためのコマンド」の行はタブ文字で始まる。複数の行を書いてもよい。タブ文字で始まらない行が来るとルールの終わりになる。
Makefile の書式には他にもいろいろと便利な記法がある。詳しくはインターネットで各自調べてほしい。

分割コンパイルのためのソースコードの書き方

複数ファイルをコンパイルするやり方は、上に述べた通り比較的簡単だが、分割コンパイルのためのソースコードを書くにはいろいろと注意が必要である。
  • 複数のファイルで使うグローバル変数や関数の 定義 (実体) は、どれかひとつのファイルだけに書く。同じ名前のグローバル変数や関数の定義が複数のファイルに存在すると、リンク時にエラーになる。同様な理由で、main 関数はプログラムにひとつしか書けない。また同じく、既存のライブラリに定義されている関数名と同じ関数名の関数を作ってはならない。
  • 別のファイルに定義のあるグローバル変数にアクセスするには、利用する行よりも前に、変数名に extern キーワードをつけて同じ型名のグローバル変数を宣言しておく。
以下、練習問題を通じて、分割コンパイルのためのソースコードの書き方を練習しよう。 基本的な手順としては、ひとつのファイルに簡単なソースコードを記述する。そして複数ファイルに分割する。その後、ファイルごとに、より複雑な処理を記述していく。慣れれば、最初から複数のファイルに別々の処理を記述すればよい。

テキスト形式とバイナリ形式の違い

プログラミング実習 IB で、テキストファイルの読み込みおよび書き込みの方法を学んだ。テキストファイルは、テキストエディタで開くことができ、人間が読める形式であるが、実はコンピュータが処理するには効率のよい形式ではない。コンピュータが処理しやすいのは、メモリに記録されるような 0 と 1 からなる数値データであり、このような形式をバイナリ形式という (バイナリとは 0 と 1 の 2 値という意味)。バイナリ形式はテキスト形式に比べて、高速に読み書きでき、ファイルサイズが小さくなりやすいため、画像データ、音楽データ、実行プログラムなど様々なデータ形式に使われている。
以下、テキスト形式 (またはアスキー形式ともいう) とバイナリ形式の違いについて、実例を見てみよう。ここでは例として、int 型の整数値 123456789 をファイルに書き込むことを考える。

テキストファイルに書き込まれる値

テキスト形式の場合は、数値を文字列として書き込む。文字列とは char 型のデータが並んだもので、char 型とは 8 ビットの整数型である。8 ビット整数値と文字の対応表である アスキーコード表 を参照して、文字が書き込まれる。つまり、テキストファイルに書かれるのは、人間が読める形式では

バイナリファイルに書き込まれる値

一方、バイナリ形式では int 型なら 4 バイトという決まったデータサイズで記録される。この値はテキストファイルのようにテキストエディタなどで見ることはできない。なぜなら、文字として読めるのは対応表の特定の 8 ビット整数値のみで、4 バイトのデータでは文字との対応がとれないためである。もしテキストエディタなどで開いてみると、無理矢理 1 バイトずつ文字と対応して表示される。このページの「事前知識: コンピュータ内部でのデータ表現」にある通り、123456789 はリトルエンディアンの CPU では

テキストファイルとバイナリファイルの読み書き

テキストファイルとバイナリファイルを読み書きするには、どちらもまずファイルをオープンする必要がある。ただしそのときのオプションの与え方が異なる。テキストファイルの場合は例えば
FILE *fin = fopen("test.txt", "r");
などと書いていた。"r" というのがオプションで「読み取り専用」という意味だった。バイナリファイルの場合はこれに "b" をつけて "rb" を指定する。オプションをまとめると次の表のようになる。
 読み取り専用書き込み専用読み書き両用 (既存ファイル対象)読み書き両用 (新しいファイルを生成)
テキスト (アスキー) 形式rwr+w+
バイナリ形式rbwbrb+wb+
ファイルを開くとき、テキスト形式のオプションを指定するとアスキーモード、バイナリ形式のオプションを指定するとバイナリモードでファイルが開かれる。これら 2 つのモードの違いは、OS による改行文字の違いを吸収するかどうかと、ファイルの終端の判定方法において異なる。テキストデータだけを読み書きするときはアスキーモード、テキストだけでなくバイナリデータも読み書きするときはバイナリモードを指定すればよい。
オープンしたファイルを読み書きするには、テキストファイルの場合は次のような関数を使っていた。
関数名動作
fscanf様々なデータをファイルから読み取る
fgets一行 (改行文字まで) テキストを読み取る
fgetc一文字だけ読み取る
fprintf様々なデータをファイルに書き込む
fputc一文字だけ書き込む
一方、バイナリファイルの場合、読み書きするのが文字列であれば上記の関数が使える (※場合によっては使えない) が、バイナリデータ、つまり文字列以外のデータの場合、専用の関数として、読み取りには fread 関数、書き込みには fwrite 関数を用いる。 
size_t fread(void *ptr, size_t size, size_t nmemb, FILE *stream);

size_t fwrite(const void *ptr, size_t size, size_t nmemb, FILE *stream);
返り値はファイルに実際に読み書きされたバイト数である。ptr は読み書きする (配列などの) 変数のアドレスである。これらの関数は、1 つあたりの大きさが size バイトの値を、stream で指定されたファイルに nmemb 個書き込む。なお、fread/fwrite 関数の呼び出しにはコストがかかるので、何回も呼ぶより、書き込む値を配列に入れて一度に書き込む方が高速である。

前提知識: PPM 画像ファイルフォーマット

今回のレポート課題では、画像を読み書きできるライブラリと、それを使ったプログラムを作成する。それに先立って、画像データというものについて解説する。画像というのは、「画像の横幅」×「画像の縦幅」の数だけ、色のついた正方形が並んでいる、という形をしている。この正方形のことを画素またはピクセル (pixel) と呼ぶ。一般に、画像データに限らず、データには決まったフォーマット (書式、形式) がある。そのフォーマットに沿ってデータを用意すれば、対応するソフトウェアがデータをうまく処理してくれる。今回のレポート課題でも、そのようなフォーマットに準拠して画像を読み書きする。
今回のレポート課題で対象とするのは、UNIX 環境で扱える簡単な画像フォーマットである PPM (Portable Pixel Map) 形式である。PPM 形式で扱えるのは、次の 3 種類の画像である。
  • 二値画像: 白か黒か、の二値だけからなる画像。
  • グレースケール画像: 白黒の濃淡で表現された画像。濃淡は 8 ビット (つまり 0〜255 の 256 段階) で表現する。0 が黒、255 が白を表す。
  • カラー画像: RGB (Red, Green, Blue) の光の三原色で表現された画像。R, G, B をそれぞれ 8 ビットで表現し、それぞれ値が大きいほど明るい。
これら 3 種類について、「画像データ」部分はアスキー形式 (テキスト形式) かバイナリ形式かを選択できる。PPM 形式の画像ファイルを開くと、次のような記述が見られる。 
(識別記号)
(画像の横幅) (画像の縦幅)
(明るさの最大値: つねに 255)
(画像データ: 値の範囲は 0 〜 255、並びの順番は左上を原点として、左から右、上から下)
「識別記号」は、上記の 3 種類 x 2 形式の計 6 通りを区別するための 2 文字の文字列である。具体的には次の表の通りである。
ヘッダ記号形式拡張子
P1二値・アスキー形式.pbm
P2グレースケール・アスキー形式.pgm
P3カラー・アスキー形式.ppm
P4二値・バイナリ形式.pbm
P5グレースケール・バイナリ形式.pgm
P6カラー・バイナリ形式.ppm

グレースケール画像の例

簡単な例として、4x4 の白黒チェッカー模様の画像を考えよう。

このグレースケール画像のアスキー形式の中身は次のようになっている。
[checker4x4_ascii.pgm]
P2
4 4
255
0 0 255 255 0 0 255 255 255 255 0 0 255 255 0 0
画像データの部分は、グレースケール画像なら画素数 (つまり「画像の横幅」×「画像の縦幅」、この場合は 4 x 4 = 16) だけ、スペースで区切られたデータが並んでいる。もしカラー画像なら、ひとつのピクセルの RGB の値が、画素ごとに並んでいる。数字の数は 3 × 「画像の横幅」×「画像の縦幅」である。 同じ内容の、バイナリ形式の画像は次のようになっている。
[checker4x4_binary.pgm]
P5
4 4
255
(※ここにバイナリデータ。(1 ピクセルあたり 1 バイト) x (4 x 4 ピクセル) = 16 バイト)
バイナリ形式では、アスキー形式と異なり、数値の間にスペースは入らず 1 画素に 1 バイトずつ連続したデータが入っている。数値の数はアスキー形式と同じである。 バイナリ形式のファイルはテキストエディタでは表示できない。バイナリ形式のファイルの中身を確認するには、例えば hexdump コマンドに -C オプションを付けて実行すればよい。 
$ hexdump -C checker4x4_binary.pgm
00000000  50 35 0a 34 20 34 0a 32  35 35 0a 00 00 ff ff 00  |P5.4 4.255......|
00000010  00 ff ff ff ff 00 00 ff  ff 00 00                 |...........|
0000001b
一番左は入力オフセット (何バイト目かを表す 16 進数)、その次以降の 2 桁の 16 進数は 1 バイト分のデータに対応する。一番右の | | で囲まれた部分は、データをアスキー表示したものである。アスキーコード表を参照すると例えば、最初の 0x50 は 'P'、0x35 は '5'、0x0a は改行文字 '\n' に対応する。

カラー画像の例

同様に、カラー画像の 4x4 チェッカー模様についても解説する。

このカラー画像のアスキー形式の中身は次のようになっている。
[color4x4_ascii.ppm]
P3
4 4
255
255 0 0 255 0 0 255 255 255 255 255 255 255 0 0 255 0 0 255 255 255 255 255 255 0 255 0 0 255 0 0 0 255 0 0 255 0 255 0 0 255 0 0 0 255 0 0 255
画像データの部分は、1 画素ごとに R, G, B の 3 つの値が並んでいる。画像の左上が原点 (0, 0) で、その値は (255, 0, 0) であり、これは赤を表す。次にバイナリ形式について、
[color4x4_binary.pgm]
P6
4 4
255
(※ここにバイナリデータ。(1 ピクセルあたり 3 バイト) x (4 x 4 ピクセル) = 48 バイト)
このファイルに対して hexdump -C で中身を表示すると 
$ hexdump -C color4x4_binary.ppm
00000000  50 36 0a 34 20 34 0a 32  35 35 0a ff 00 00 ff 00  |P6.4 4.255......|
00000010  00 ff ff ff ff ff ff ff  00 00 ff 00 00 ff ff ff  |................|
00000020  ff ff ff 00 ff 00 00 ff  00 00 00 ff 00 00 ff 00  |................|
00000030  ff 00 00 ff 00 00 00 ff  00 00 ff 0a              |............|
0000003c
アスキーコード表を参照すると、32 35 35 0a がアスキー表示で 255\n に対応し、それ以降の ff 00 00 が、原点 (0, 0) における RGB の値 (255, 0, 0) に対応する。

画像の閲覧方法

PGM や PPM 形式の画像を閲覧するには、UNIX の display コマンドを使うのがひとつの方法である。 
$ display  checker4x4_ascii.pgm
もうひとつは、convert コマンドで画像のフォーマットを bmp や jpg などに変換する方法である。例えば jpg なら、Web ブラウザを始めいろいろなソフトウェアで画像を閲覧できる。 
$ convert  checker4x4_ascii.pgm  checker4x4.jpg
逆に、convert コマンドで、jpg 形式などから PPM 形式に変換できる。 
$ convert  test.jpg  test_gray.pgm    ←拡張子で画像の種類が判定されている…pgm だとグレースケール画像

$ convert  test.jpg  test_color.ppm    ←拡張子で画像の種類が判定されている…ppm だとカラー画像
なお、注意として、convert コマンドではバイナリ形式の PPM ファイルが出力される。つまり、「識別記号」の文字列は、P4, P5, P6 のいずれかになる。
※注意: エラーチェックのコード、例えばファイルをちゃんと開けたかどうか、期待する識別文字 (P2, P5 など) を読み取れたかどうか、などを面倒臭がらずに書くこと。そうしないとバグが起きた際にどこに問題があるか特定するのに時間がかかる。
